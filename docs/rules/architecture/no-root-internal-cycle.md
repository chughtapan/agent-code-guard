# `agent-code-guard/no-root-internal-cycle`

**What it flags:** Root/public-layer files and `internal/` files importing
through each other. Specifically: any path where an `internal/*` file imports
from the package root (e.g., `src/index.ts`) AND the root re-exports from
that internal file.

**Why:** `internal/` exists to hide implementation decisions from the
public/root surface. If internal code imports the root facade, the private
layer depends on the public layer that is supposed to hide it — the
abstraction has inverted. New consumers to the root then transitively pull
in everything internal.

## Before (flagged)

```ts
// src/index.ts (public root)
export const VERSION = "1.0.0";
export { makeClient } from "./internal/client";

// src/internal/client.ts
import { VERSION } from "../index";  // internal depends on root → flagged

export function makeClient() {
  return { version: VERSION };
}
```

The internal client now needs the root to load before it can resolve. Bundlers
have to defer resolution; tests have to mock the root. The "private" layer
has become a consumer of the public one.

## After (preferred) — pull shared values below both layers

```ts
// src/version.ts
export const VERSION = "1.0.0";

// src/index.ts (public root)
export { VERSION } from "./version";
export { makeClient } from "./internal/client";

// src/internal/client.ts
import { VERSION } from "../version";  // both layers depend on shared kernel
```

Now `version.ts` is below both `index.ts` and `internal/client.ts`. The
dependency direction is one-way again.

## After (preferred) — inject from the entrypoint

If the shared value isn't a constant (e.g., it's config, a logger, or a
client), inject it from the composition root:

```ts
// src/index.ts
import { makeClient } from "./internal/client";

export const VERSION = "1.0.0";
export const client = makeClient({ version: VERSION });

// src/internal/client.ts (no import from root)
export function makeClient(config: { readonly version: string }) {
  return { version: config.version };
}
```

`internal/client.ts` doesn't know where its `version` comes from — that's the
caller's job.

## Options

This rule has no tunable options. The check is structural: any
internal-imports-root edge fails.

## Suppressing per-file via a directive

Root↔internal cycles are almost always real bugs. Suppression should be
very rare. If you genuinely need it (e.g., a generated re-export that has
to reference both layers), suppress the rule for the offending file with a
written reason:

```ts
// @agent-code-guard/architecture-exception: no-root-internal-cycle
// reason: generated by codegen; cycle resolved at build time

import { type Plugin } from "../index";
```

The `reason:` line is required; missing reasons surface as
`architecture-directive-parse-error` diagnostics.

If you find yourself directive-suppressing this rule on hand-written code,
the design is upside-down. Either move the shared value below both layers
or invert with dependency injection.

## Rationale

The whole point of an `internal/` folder is hiding implementation. A cycle
between root and internal proves the hiding has failed.
